SIMPLER MANAGEMENT



LOG FILE PREFIX AND RETENTION OPTIONS

NEW DBMS_CLOUD.COPY_DATA supports the format parameter options for managing log retention and log prefix


SIMPLER MANAGEMENT


KEY SALES PLAYS

DATA WAREHOUSING
AND DATA LAKE

MODERNIZE WITH
DATABASE 19c

CLOUD-NATIVE APPS

OVERVIEW
Two additional parameters have been added to DBMS_CLOUD.COPY_DATA to manage prefixes and retention periods for the logfile and badfile tables, which are generated during data loading.

The "logprefix" attribute specifies a string value that determines the prefix for the logfile_table and badfile_table files. The log table name format is logprefix$operation_id. For example if you want to prefix the logfile and badfile tables with the word 'TEST' the using the following:

format => JSON_OBJECT ('logprefix' value 'TEST')

will result in logfile and badfile tables with the following names, assuming the workload identifier is 2: TEST$2_LOG and TEST$2_BAD.

The "logretention" attribute specifies a positive integer duration, in days, for which the logfile_table and badfile_table files are retained: For example the following sets the retention period to 7 days:

format => JSON_OBJECT ('logretention' value 7)

BUSINESS BENEFIT
Gives users more control so they can assign their own naming convention to the logfile and badfile tables created by DBMS_CLOUD.COPY_DATE and set the retention period.

SIMPLIFIED WALLETS FOR CROSS-REGION AUTONOMOUS DATA GUARD

NEW - Change to the connect strings in the wallet when using cross-region Autonomous Data Guard.


SIMPLER MANAGEMENT

KEY SALES PLAYS

DATA WAREHOUSING
AND DATA LAKE

MODERNIZE WITH
DATABASE 19c

CLOUD-NATIVE APPS

OVERVIEW
When you enable Autonomous Data Guard with a cross-region (remote) standby database, the wallet and connection string from the primary database will contain only the hostname of the primary database, and the wallet and connection string from the remote database will contain only the hostname of the remote database. This applies to both instance and regional wallets.

BUSINESS BENEFIT
Oracle recommends configuring applications running in the primary region to use the wallet or connection string downloaded from the primary database and configuring the applications running in the remote region to use the wallet or connection string downloaded from the remote database.

This simplifies the set and configuration process for applications where Autonomous Data Guard is used with the standby in a different region.

IMPROVEMENTS TO DBMS_CLOUD_REPO PACKAGE

Support for (Git) repository branch management and schema export and install


AGILE DEVELOPMENT

KEY SALES PLAYS

DATA WAREHOUSING
AND DATA LAKE

MODERNIZE WITH
DATABASE 19c

CLOUD-NATIVE APPS


OVERVIEW

Create and Manage Branches
The DBMS_CLOUD_REPO management routines allow you to manage Cloud Code Repository branches by creating, listing, merging, or deleting branches in a repository. It is possible to list commits in a branch in Cloud Code Repository based on the following: repository, branch, the file path of the branch, and SHA/commit_id.

Export Schema Objects
The DBMS_CLOUD_REPO management routine allows you to export metadata of the objects in a schema to the Cloud Code Repository branch. You can filter your list based on the object names or object types.

SQL Install Operations
The DBMS_CLOUD_REPO SQL Install operations allow you to store and download SQL scripts from a Cloud Code Repository.

CONNECTING TO MULTIPLE INSTANCES OF OCI DATA CATALOG

Connect multiple OCI Data Catalog instances to Autonomous Database
DATA LAKE

KEY SALES PLAYS

DATA WAREHOUSING
AND DATA LAKE

MODERNIZE WITH
DATABASE 19c

OVERVIEW
Instead of each Autonomous Database instance supporting a single connection to a Data Catalog instance, this feature allows connections to multiple Data Catalog instances.

BUSINESS VALUE
Query External Data with Data Catalog Oracle Cloud Infrastructure Data Catalog is the metadata management service for Oracle Cloud that helps you discover data and support data governance. It provides an inventory of assets, a business glossary, and a common metastore for data lakes.

Autonomous Database can leverage this metadata to dramatically simplify management for access to your data lake's object store. Instead of manually defining external tables to access your data lake, use the external tables that are defined and managed automatically. These tables will be found in Autonomous Database protected schemas that are kept up to date with changes in Data Catalog.
