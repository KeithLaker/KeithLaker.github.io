#1 SIMPLER LICENSING


BYOL LICENSING

UPDATE - Licensing rules for BYOL have been changed.


SIMPLER LICENSING

CHALLENGE
Customers wishing had to have Multitenant licences along with Database licences to use BYOL with ADB.

SOLUTION
Multitenant requirement for BYOL with ADB is now lifted. Customers do not need to bring in Multitenant licenses anymore for BYOL with Autonomous Database.

Obviously, the other licensing requirements still apply: For 17 OCPUs or more of Oracle Autonomous Database customers must have Real Application Clusters licenses and Active Data Guard is required if the customer wants to enable Autonomous Data Guard.

#2 STRONGER SECURITY


AZURE ACTIVE DIRECTORY

NEW - Operations Insights: SQL Explorer


STRONGER SECURITY

CHALLENGE
Many on-premises customers want to integrate their Oracle Database with Microsoft Active Directory to streamline user management and help reduce risk. As these customers migrate to the cloud, many are looking to centralize identities in their cloud directories while their Oracle Database runs in Oracle Cloud Infrastructure (OCI).

SOLUTION
Oracle Cloud Infrastructure (OCI) Autonomous Database (Shared Infrastructure) users can be centrally managed in Microsoft Azure Active Directory (Azure AD) for stronger security instead of locally in every database. Azure AD users and applications can use Azure AD OAuth2 access tokens to access the Autonomous Database. Azure AD users don't have to remember separate database credentials and just use their existing Azure AD SSO credentials.




LEARN MORE

DOC: Use Azure Active Directory (Azure AD) with Autonomous Database, click here
DOC: Authenticating and Authorizing Microsoft Azure Active Directory Users for Oracle Autonomous Databases, click here
BLOG: Oracle Autonomous Database users can now be authenticated and managed with Azure Active Directory, click here


BUSINESS VALUE


Direct integration with Azure AD simplifies the user authentication experience. Azure users can use their SSO credentials to authenticate to the OCI database directly without managing their identity profile in the federated identity service. Since the database can work directly with Azure AD tokens, there is no need for identity administrators to manage federation between cloud identity providers.
