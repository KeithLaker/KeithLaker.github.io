Overview of Oracle's zero-downtime patching



Automated Patching Process - why is it important?

Autonomous Database instances are automatically patched on a regular bi-weekly cycle. The patches include new features, bug fixes, and security patches. By automating the patching process Oracle can ensure that your databases are always up-to-date with the latest security patches, which helps to protect your data from unauthorized access.

Automated patching is important for the following key reasons:

Security: Fully automated patching transparently fixes vulnerabilities that might be susceptible to cyberattacks, helping your organization reduce its security risk.

Database uptime: Fully automated patching ensures all software (database, O/S, storage servers etc) is kept up-to-date and runs smoothly, supporting system uptime requirements.

Compliance: With the continued rise in cyber-attacks, organizations are often required by regulatory bodies to maintain a certain level of compliance. Patch management is a necessary piece of adhering to compliance standards.

Feature and enhancements: Patching typically goes beyond delivering fixes and includes new feature/functionality updates. ADB Patches are critical to ensuring that you have the latest and greatest the database has to offer.


Oracle's Security-First Mindset

Securing your data is essential. Oracle's automated end-to-end patching process ensures instances are up-to-date with the latest security patches. This helps to protect your data from known security vulnerabilities and prevents attackers from exploiting them.


Backed by a Zero-Regression SLO

Oracle provides a service level objective of zero regressions in your production database due to these patches.

After a patch is applied to a database on the Early patch level, if an issue is found and reported through a service request, Oracle will use commercially reasonable efforts to address the problem so that the same issue does not occur when the patch is applied to the production database.



Is there any need to test upcoming patches?



Why test patches?

Even with the extreme levels of automated monitoring and testing along with the decades if best practices used by Autonomous Database there is no way to provide a 100% guaranteed that a patchset is ever going to be completely bug free. That's why we have a pre-release patching cycle update that provides 7-days advanced access to an upcoming patchset so cusomters can run and validate their workloads.

The high level aim of patch testing is to determine if a patch will cause any problems for a specific workload. The best way to do so is to create a test environment that mirrors the production environment. In the past this has been seen as an expensive operation because it entailed building and maintaining a duplicate real-world environment. Due to operational complexities and costs, most IT teams have resorted to using smaller data sets and sample production workloads for testing which provided only very limited reassurances.


What's involved patch testing?

Patch testing involves more than just ensuring that your database is up and running and your tools can connect. Typical best practices recommend running running smoke tests to determine if all the main functions of a workload appear to operate correctly.

Tests involve comparing application performance before and after patch deployment and ensuring that other applications running on the target environment are not negatively affected. It requires simulating usage cycles and seasonal peak workloads because some problems only appear above a certain threshold or when certain functions are executed -- monthly payroll, for example.



How ADB simplifies and automates patch testing?



Get early access to upcoming patches

When you provision or clone an Autonomous Database instance you can select a patch level to apply to upcoming patches. There are two patch level options: Regular and Early.

Provisioning or cloning an instance and setting the patch level to Early allows you to use and to test upcoming patches before they are applied to all systems.


Caputure production workloads using Real Application Testing

Oracle's unique Real Application Testing provides an extremely cost-effective and easy-to-use proactive management solution that enables businesses teams to quickly and easily fully assess the outcome of an planned patching cycle.

Real Application Testing captures a workload on the production system and then replays it on a separate test system with the exact timing, concurrency, and transaction characteristics of the original workload.

This allows teams to test the effects of the next patchset on a workload without affecting the production system. It provides a simple and accurate method for patchset testing.
